# Wireline
### 一个别名，连接一切

版本 v0.1 草案 | 2026 年 7 月

---

## 一、项目概述

Wireline 是一款面向 macOS 的原生 SSH 客户端，核心理念是把"一个短别名连接一切服务器"的效率体验，从命令行搬进一个真正的图形化客户端。它不是又一个通用远程终端工具，而是专注于解决工程师在管理数十台服务器时最真实的痛点：别名难记、密钥密码混用、登录后重复操作、批量运维缺乏统一入口、换设备迁移麻烦。

与市面上多数 SSH 客户端不同，Wireline 的主机数据始终以标准 `~/.ssh/config` 作为唯一数据源，不发明私有数据库或云同步格式——图形界面只是这份标准配置的一层高效外壳。这意味着即便某天不再使用 Wireline，命令行 `ssh`、`scp`、`rsync`、VS Code Remote 依然能直接读取同一份配置，用户永远不会被工具锁定。

### 1.1 命名释义

Wireline 取自通信与工程领域中"线路"的意象：一端是本地终端，一端是分布在各地的服务器，中间是一条被简化、被管理、被信任的连接线路。名字同时呼应了工具底层依赖的标准协议特性——不改变线路本身，只让线路更容易被找到和使用。

### 1.2 平台范围

Wireline 定位为 macOS 原生应用，充分利用 macOS 平台能力（Keychain、Spotlight 式搜索交互、菜单栏常驻、系统级安全存储），不追求跨平台一致性。这一取舍让产品可以把体验做深、做窄，而不是为了兼容多平台而牺牲原生体验。

## 二、背景与痛点

随着基础设施规模扩大，工程师日常面对的服务器数量往往从个位数迅速增长到数十台，分布在不同云厂商、不同地域、不同业务线。原生 `ssh` 命令或传统终端在这种规模下暴露出明显的效率问题：

- IP、端口、用户名难以记忆，别名管理依赖零散的备忘录或历史命令
- 密钥登录与密码登录混用，登录方式需要人工判断
- 部分主机登录后还需手动执行 `sudo -i` 等重复操作
- 跨设备迁移时，主机清单与密码难以安全、完整地随身携带
- 面对一批主机的巡检、变更、端口转发等运维动作缺乏统一入口
- 命令行工具虽然高效，但缺乏可视化的分组、状态一览和低门槛的图形交互，对多人协作场景不友好

这些问题并非需要一个全新的、重量级的堡垒机系统来解决——它们本质上是对 SSH 使用体验的增强诉求。Wireline 的定位正是把命令行时代验证过的高效范式（别名、分组、批量、安全存储），用原生客户端的形式重新呈现，做到"命令行的效率 + 图形界面的直观"。

## 三、目标用户与使用场景

### 3.1 目标用户

- 使用 macOS 作为主力工作设备、独立管理数台到数十台服务器的基础设施/运维/后端工程师
- 需要频繁在多地域节点（如香港、新加坡、内地）之间切换的团队
- 希望在办公设备与个人设备之间快速、安全迁移主机清单的工程师
- 对命令行有一定熟悉度，但更偏好图形化状态总览与鼠标/快捷键混合操作的用户

### 3.2 典型场景

- **日常连接**：呼出全局搜索窗口，输入别名首字母即可连接，密钥/密码自动识别
- **状态总览**：客户端主界面按分组展示所有主机及在线状态，无需逐台 ping
- **批量运维**：在客户端中框选一组主机，对其并发执行同一条命令并查看聚合结果
- **内网访问**：通过图形化配置将远端服务端口映射到本地，支持借跳板机访问内网资源
- **设备迁移**：导出一份加密备份文件，在新 Mac 上一次导入即可恢复全部主机与密码
- **登录后自动化**：为特定主机标记"登录后自动 sudo"，减少重复操作

## 四、产品定位与设计原则

Wireline 定位为"macOS 上的高效 SSH 连接与轻量运维客户端"，而非替代堡垒机或企业级零信任网关的重型平台。其设计遵循以下原则：

- **标准优先**：数据始终落在 `~/.ssh/config`，图形界面只是标准配置之上的高效外壳，保证可移植性与可退出性
- **原生体验**：深度使用 macOS 平台能力（Keychain、菜单栏、全局快捷键、系统通知），而非跨平台框架的通用界面
- **安全默认**：密码类凭据默认交由 macOS Keychain 加密存储，绝不以明文形式落盘
- **渐进增强**：核心连接体验（打开即连）保持极简，进阶能力（批量、转发、备份）通过侧边栏和面板按需展开，不强加认知负担

## 五、核心功能

| 功能模块 | 能力说明 | 对应痛点 |
|---|---|---|
| 全局快捷连接 | 类似 Spotlight 的呼出式搜索窗口，模糊匹配别名/描述/分组，回车即连，自动识别密钥或密码登录 | 服务器规模增长后 IP/账号难以记忆，切换终端繁琐 |
| 主机总览面板 | 按分组以列表/卡片展示全部主机，实时显示在线状态、认证方式、所属分组 | 缺乏统一的主机健康检查与结构化视图 |
| 分组与标签 | 拖拽或批量操作对主机分组，支持按地域、业务线自由归类 | 主机数量多后缺乏结构化组织 |
| Keychain 密码存储 | 密码类凭据交由 macOS Keychain 加密存储，配置文件中仅保留认证方式标记，不落明文 | 明文密码存在配置文件中的安全隐患 |
| 加密备份与迁移 | 一键导出全部主机与 Keychain 密码为加密文件，新 Mac 上一键导入恢复 | 更换办公设备时凭据迁移繁琐易出错 |
| 登录后自动提权 | 为账号标记"自动 sudo"，登录后自动复用密码完成提权，无需手动输入 | 部分主机登录后需手动切换 root，操作重复 |
| 批量命令执行 | 多选或按分组/通配符选择一组主机，在客户端内并发执行同一命令并查看聚合输出 | 集群或多地节点巡检、变更操作低效 |
| 图形化端口转发 | 在界面中配置本地端口与远端服务的映射关系，支持借跳板机访问内网资源，一键启停 | 访问内网服务/数据库需要手动维护转发规则 |
| 连通性检查 | 后台定时或手动触发，按分组或全量检测主机在线状态并在总览面板高亮 | 缺乏统一的主机健康检查手段 |
| 标准配置兼容 | 底层始终读写标准 `~/.ssh/config`，客户端内的别名对终端里的 ssh/scp/rsync/VS Code Remote 全部生效 | 担心工具锁定、迁移成本高 |

## 六、技术架构

Wireline 采用原生 macOS 应用架构，核心分为四层：

- **界面层**：SwiftUI 构建主窗口（主机总览、分组侧边栏、批量执行面板）与菜单栏常驻入口，全局快捷键唤起快捷连接搜索窗
- **配置层**：解析并回写标准 OpenSSH client config（`~/.ssh/config`）；工具自身需要的额外元数据（分组、描述、认证方式、是否自动提权）以注释行形式写入同一个 Host 块中，保证原生 ssh 工具忽略这些注释、正常解析
- **凭据层**：密码类凭据通过 macOS Keychain Services API 加密存储，客户端仅持有指向 Keychain 条目的引用
- **连接层**：通过系统 ssh 二进制或嵌入式 SSH 库建立会话，终端渲染可选择内置终端视图或调用用户偏好的外部终端（Terminal.app / iTerm2）
- **备份层**：导出/导入功能基于对称加密，将主机清单与 Keychain 中对应密码一并打包为单一加密文件，实现跨设备安全迁移

这一架构决定了 Wireline 没有服务端、没有中心化数据库，所有状态都在用户本机，天然规避了云端凭据托管带来的合规与信任问题，这也与 Longbridge 内部对敏感凭据"不出本机"的一贯要求相符。

## 八、数据安全与隐私

- 密码类凭据仅存储于 macOS Keychain，不以明文形式出现在任何配置文件中
- 加密备份文件采用口令派生密钥加密，口令不落盘、不上传，丢失口令等同于备份不可恢复
- 客户端不产生任何网络回传行为，所有操作均在本机与目标服务器之间直接发生
- 批量执行、端口转发等高权限操作均基于用户已有的 SSH 凭据，不引入额外的信任链
- 应用本身不托管任何服务器数据，卸载客户端后 `~/.ssh/config` 中的主机记录依然完整可用

## 十、风险与应对

- **原生开发投入更高**：相比跨平台方案，纯原生 macOS 客户端在人力投入上更集中，需要 Swift/SwiftUI 相关能力 —— 建议优先复用已有脚本工具（s）验证过的交互逻辑，降低产品设计风险
- **单机备份文件是高价值目标**：一旦备份文件与口令同时泄露，风险等同于全部服务器凭据泄露 —— 后续可引入备份文件的二次因素保护或有效期限制
- **从个人工具走向团队协作的复杂度**：分组、批量能力是为个人设计的，团队共享需要额外的权限与审计层，规划在 V2.0 中审慎引入，避免过早引入不必要的重量级功能
- **仅支持 macOS 带来的用户覆盖限制**：明确产品定位、优先服务好核心用户群体，不因追求覆盖面而稀释原生体验

## 十一、总结

Wireline 把命令行时代已经验证过的高效范式——别名连接、分组管理、安全存储、批量运维——重新设计为一款原生 macOS SSH 客户端。它既保留了标准 `~/.ssh/config` 带来的可移植性与可退出性，又用图形界面降低了操作门槛、提升了多主机场景下的状态感知能力。下一步的产品化重点，是在保持"标准优先、原生体验"这一核心气质的前提下，把细节体验打磨到位，并为团队协作场景设计出不破坏个人使用体验的扩展路径。
